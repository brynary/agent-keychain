# `agent-keychain` Implementation Handoff

## Overview

`agent-keychain` is a macOS Swift CLI for managing local credentials and browser sessions used by AI agents.

The tool has two responsibilities:

1. **Small secrets**: API tokens, service credentials, disk-image passwords, and other sensitive values stored in the selected project keychain storage backend.
2. **Large encrypted state**: Browser profiles and other mutable session state stored inside encrypted APFS sparsebundle volumes.

The tool is **project-scoped**, not user-global. Each project has its own configuration, keychain, encrypted volumes, browser profiles, and audit log.

## Motivation

AI agents often receive access to apps through:

1. API tokens stored in `.env` files.
2. Authenticated browser sessions created by logging into Chrome and letting the agent reuse that session.

This creates several risks:

* **Credential theft**: local programs, dependencies, package scripts, or malicious agent actions may read `.env` files or browser profile data.
* **Privilege escalation**: an agent may use a credential or browser session that was not intended for the current task.
* **Lack of auditability**: it is difficult to reconstruct when a credential was used, which browser profile was opened, or why a privileged session was unlocked.
* **Over-broad browser access**: if an agent uses the human’s normal Chrome profile, it inherits personal/admin authority across many apps.

`agent-keychain` provides a local security boundary for agent credentials and browser state:

* No plaintext `.env` files.
* No agent use of the user’s normal Chrome profile.
* Browser profiles encrypted at rest.
* Touch ID required to unlock sensitive resources.
* Role-based separation of duties, such as `regular`, `workspace-admin`, and `finance`.
* A separate physical keychain per project when supported, with an explicit fallback keychain mode when modern macOS APIs cannot support the physical-keychain approach.
* Project-local audit logs for every secret read, volume unlock, browser launch, policy mutation, and privileged action request.

This does **not** fully solve runtime compromise. Once a volume is mounted or a secret is injected into a child process, that data is available to processes running as the user. The tool is an at-rest protection, workflow isolation, and auditability layer. Future work may add brokered tools so agents never receive raw privileged credentials.

# Core Concepts

## Project

A **project** is the top-level unit of isolation.

Each project has its own:

* config file
* keychain storage backend, preferably a physical project keychain file
* secrets
* APFS sparsebundle volumes
* browser profiles
* audit log

The project lives inside the repository or workspace that uses it.

Recommended project directory:

```text
<project-root>/
  .agent-keychain/
    config.json
    config.integrity.json
    audit.jsonl
    locks/
    keychains/
      project.keychain-db
    volumes/
      RegularBrowser.sparsebundle
      WorkspaceAdminBrowser.sparsebundle
      FinanceBrowser.sparsebundle
```

The `.agent-keychain` directory should be treated as local machine state, not committed to Git.

Recommended `.gitignore` entries:

```gitignore
.agent-keychain/keychains/
.agent-keychain/volumes/
.agent-keychain/audit.jsonl
.agent-keychain/config.integrity.json
.agent-keychain/locks/
```

Whether `config.json` is committed is a project decision. For most projects, treat it as local config first. A future version may support a committed template config.

## Role

A **role** is a compartment of agent authority within a project.

Examples:

* `regular`: day-to-day low-risk agent work.
* `workspace-admin`: Google Workspace / identity administration.
* `finance`: banking, billing, payroll, vendor payments, and other money movement.
* `production`: infrastructure and production operations.
* `legal`: contracts and confidential legal documents.

Each role owns its own:

* secrets
* encrypted volumes
* browser profiles
* default timeout policy
* audit policy
* reason/approval requirements

Roles are not hierarchical. `finance` is not a subtype of `admin`; it is a separate authority domain.

## Secret

A **secret** is a small sensitive value stored in the project’s dedicated keychain.

Examples:

* `github-readonly`
* `github-pr-writer`
* `google-workspace-admin`
* `mercury-api-key`
* `stripe-readonly`

Each secret belongs to exactly one role.

Secret values must never be stored in the project config file. Only metadata should be stored there.

## Volume

A **volume** is an encrypted APFS sparsebundle managed by `agent-keychain`.

Volumes are used for large mutable state such as browser profiles.

Each volume belongs to exactly one role.

Example volumes:

* `RegularBrowser`
* `WorkspaceAdminBrowser`
* `FinanceBrowser`

Each encrypted APFS sparsebundle password should be randomly generated by `agent-keychain`, stored in the selected keychain storage backend, and retrieved only after Touch ID / user-presence authentication.

## Browser Profile

A **browser profile** is a Chrome user data directory stored inside a role-owned encrypted APFS volume.

Example browser profiles:

* `GitHub`
* `Linear`
* `GoogleWorkspaceAdmin`
* `Mercury`
* `Ramp`
* `StripeDashboard`

Browser profiles must never use the user’s normal Chrome profile under:

```text
~/Library/Application Support/Google/Chrome
```

Instead, Chrome must be launched with `--user-data-dir` pointing inside a mounted encrypted APFS volume.

Important limitation: the sparsebundle protects the Chrome user data directory at rest, but it does not guarantee that every browser-related secret lives inside the sparsebundle. Chrome and macOS may store supporting auth material in the user's keychains, including Chrome Safe Storage data, passkeys, or sync-related state. `agent-keychain` should describe browser profile encryption as protection for the managed user data directory, not as complete containment of all Chrome authentication material. For privileged profiles, prefer low-privilege app accounts and disable browser sync, password storage, and passkey use where practical.

# Project Layout

## Default Layout

From a project root:

```text
.agent-keychain/
  config.json
  config.integrity.json
  audit.jsonl
  locks/
  keychains/
    project.keychain-db
  volumes/
    RegularBrowser.sparsebundle
    WorkspaceAdminBrowser.sparsebundle
    FinanceBrowser.sparsebundle
```

Mounted volumes should use stable project-specific mountpoints:

```text
/Volumes/AgentKeychain-<project-name>-RegularBrowser
/Volumes/AgentKeychain-<project-name>-WorkspaceAdminBrowser
/Volumes/AgentKeychain-<project-name>-FinanceBrowser
```

If the project name contains spaces or special characters, sanitize it for mountpoint use.

## Project Initialization

Command:

```bash
agent-keychain init
```

Behavior:

1. Detect project root.
2. Create `.agent-keychain/`.
3. Create `.agent-keychain/locks/`.
4. Create `.agent-keychain/audit.jsonl`.
5. Create `.agent-keychain/config.json` with empty roles, secrets, volumes, and browser profiles.
6. Select the keychain storage backend.
7. Prefer creating `.agent-keychain/keychains/project.keychain-db`.
8. If the physical project keychain is unavailable on this host, enter explicit fallback keychain mode and record that mode in config, status output, and the audit log.
9. Create `.agent-keychain/config.integrity.json` from the initial canonical config hash.
10. Prompt the user to confirm keychain storage backend and password handling as applicable.

In physical-keychain mode, because we want Touch ID convenience and do not want the user remembering another password, the project keychain password should be generated randomly and stored in the user’s login keychain with `userPresence` access control.

This creates two layers:

```text
Touch ID
  -> releases project keychain password from login keychain
  -> unlocks project.keychain-db
  -> reads project secrets / volume passwords from project.keychain-db
```

In physical-keychain mode, the project keychain itself stores the project secrets. The login keychain stores only the generated password needed to unlock the project keychain. In fallback mode, project secrets and volume passwords are stored directly in the login keychain under the project-scoped fallback namespace.

# Project Keychain Storage

## Primary Mode

Prefer a **separate physical keychain**:

```text
.agent-keychain/keychains/project.keychain-db
```

In physical-keychain mode, this project keychain stores:

* API tokens
* service credentials
* APFS sparsebundle passwords
* other project-specific agent secrets

In physical-keychain mode, the user should not have to remember the project keychain password.

## Fallback Mode

If the macOS compatibility spike or host runtime checks show that a separate physical keychain is not viable, v0.1 may use an explicit fallback mode.

Fallback mode stores the same logical project items in the user's login keychain under the project keychain namespace, with each sensitive item protected by `userPresence`. Fallback mode is acceptable v0.1 behavior only when:

* the project config records that fallback mode is active
* `agent-keychain status` reports that physical project-keychain separation is unavailable
* the audit log records the fallback-mode decision during `init`
* service names include the project identity and role/resource names
* secret values and sparsebundle passwords still never live in config, process arguments, or logs

## Project Keychain Password

At `agent-keychain init` in physical-keychain mode:

1. Generate a random high-entropy project keychain password.
2. Create `.agent-keychain/keychains/project.keychain-db`.
3. Store the project keychain password in the user’s login keychain.
4. Protect that login-keychain item with `userPresence`.
5. Unlock the project keychain only after retrieving the project keychain password via Touch ID.

At `agent-keychain init` in fallback mode:

1. Generate no project keychain password.
2. Store each project secret and volume password directly in the login keychain using the project-scoped fallback namespace.
3. Protect each sensitive fallback item with `userPresence`.
4. Record fallback mode in config, status, and audit output.

Login keychain item:

```text
service: agent-keychain.project.<project-name>.keychain-password
account: default
value: <random project keychain password>
access: userPresence
```

Project keychain items:

```text
service: agent-keychain.role.<role>.secret.<secret-name>
account: default
value: <secret value>
```

```text
service: agent-keychain.role.<role>.volume.<volume-name>.password
account: default
value: <random sparsebundle password>
```

## Keychain Unlock Flow

For any operation requiring project secrets in physical-keychain mode:

```text
1. Read project keychain password from login keychain.
2. This triggers Touch ID / user-presence.
3. Unlock .agent-keychain/keychains/project.keychain-db.
4. Read the requested secret or volume password from the project keychain.
5. Optionally lock the project keychain again after use.
```

For any operation requiring project secrets in fallback mode:

```text
1. Read the requested project-scoped fallback item from the login keychain.
2. This triggers Touch ID / user-presence for that item.
3. Use the secret or volume password for the requested operation.
```

## Project Keychain Timeout

In physical-keychain mode, the project keychain should support timeout locking.

Default behavior:

* unlock for the operation
* read/write needed item
* lock on command exit

For longer operations such as `run` or `browser open`, support:

```bash
agent-keychain run --keychain-timeout 5m ...
```

In physical-keychain mode, the project keychain should be locked on exit via a `defer` / cleanup handler.

## macOS Keychain Compatibility Spike

Before implementation begins, run a short compatibility spike for the dedicated project keychain approach.

The spike must verify:

* creating and unlocking `.agent-keychain/keychains/project.keychain-db` on the supported macOS versions
* storing and reading generic password items in the project keychain
* `SecItemAdd`, `SecItemCopyMatching`, and `SecItemDelete` behavior with a non-login keychain
* whether project-keychain items require explicit search-list or keychain-reference handling
* whether per-item `userPresence` works reliably in the project keychain
* whether the `/usr/bin/security` CLI can provide a reliable fallback for create, unlock, lock, add, read, and delete operations

If custom physical keychains are not viable with modern macOS APIs, keep the v0.1 security model intact by falling back to a login-keychain storage namespace that includes the project identity and role/resource names, with all sensitive items protected by `userPresence`. In that fallback mode, update the project status output and audit log to state that physical project-keychain separation is unavailable on the current host.

Spike result on the current host:

* `/usr/bin/security` can create, unlock, add generic password items, read generic password items, delete generic password items, and lock a throwaway non-login keychain.
* Swift Security APIs can create and unlock a throwaway non-login keychain with `SecKeychainCreate` and `SecKeychainUnlock`.
* `SecItemAdd` can write generic password items to the non-login keychain when passed the explicit keychain reference.
* `SecItemCopyMatching` can read those items when passed an explicit `kSecMatchSearchList`.
* `SecItemDelete` can delete those items when passed an explicit `kSecMatchSearchList`.
* These APIs work but produce deprecation warnings for `SecKeychain` APIs. v0.1 may still use them, with `/usr/bin/security` as fallback.
* `SecItemAdd` with `kSecAttrAccessControl` and `.userPresence` against the non-login keychain failed with `errSecParam`. v0.1 must not rely on per-item `userPresence` inside the physical project keychain.

Conclusion for this host: use physical-keychain mode for v0.1. Keep fallback mode in the design for hosts where the same compatibility checks fail.

# Keychain Requirements

## Per-Item Access Control

The login-keychain item that stores the project keychain password must use:

```swift
kSecAttrAccessibleWhenUnlockedThisDeviceOnly
```

and:

```swift
.userPresence
```

Use a fresh `LAContext` per sensitive read.

Inside the project keychain, per-item Touch ID is not required for v0.1. The current host spike found that adding a `.userPresence` access-control item to a non-login keychain fails with `errSecParam`. The required security boundary in physical-keychain mode is:

```text
Touch ID gates unlocking the project keychain.
The project keychain physically separates project secrets from the user’s login keychain.
```

Enforce Touch ID at project keychain unlock time by retrieving the generated project keychain password from the login keychain with `userPresence`.

In fallback mode, there is no physical separation from the login keychain. The required boundary is project-scoped item naming plus per-item `userPresence` on each sensitive item.

## `--touch-id` Flag Semantics

The `--touch-id` flag marks a secret or volume password as requiring user-presence before use.

In physical-keychain mode, `--touch-id` does not mean per-item Touch ID inside `project.keychain-db`. v0.1 gates access by requiring Touch ID to retrieve the project keychain password from the login keychain, then unlocking the physical project keychain for the operation.

In fallback mode, `--touch-id` means the fallback login-keychain item itself must be stored with per-item `userPresence`.

## Secret Keychain Item Naming

Use one item per secret:

```text
service: agent-keychain.role.<role>.secret.<secret-name>
account: default
value: <secret value>
```

In physical-keychain mode, these service names live inside the physical project keychain. In fallback mode, prefix service names with the project identity so fallback items are project-scoped inside the login keychain.

Examples:

```text
agent-keychain.role.regular.secret.github-readonly
agent-keychain.role.workspace-admin.secret.google-workspace-admin
agent-keychain.role.finance.secret.mercury-api-key
```

## Volume Password Keychain Item Naming

Use one item per volume password:

```text
service: agent-keychain.role.<role>.volume.<volume-name>.password
account: default
value: <random sparsebundle password>
```

In physical-keychain mode, these service names live inside the physical project keychain. In fallback mode, prefix service names with the project identity so fallback items are project-scoped inside the login keychain.

Examples:

```text
agent-keychain.role.regular.volume.RegularBrowser.password
agent-keychain.role.workspace-admin.volume.WorkspaceAdminBrowser.password
agent-keychain.role.finance.volume.FinanceBrowser.password
```

## Secret Generation

For APFS sparsebundle passwords and the physical-mode project keychain password:

* Do not ask the user to invent or remember passwords.
* Generate high-entropy random passwords.
* Store them only in Keychain.
* Never log, print, or store the values in config.

Suggested implementation:

* Generate at least 32 random bytes.
* Encode as base64url or similar printable form.

# APFS Volume Requirements

## Creating a Volume

Use `hdiutil create` to create encrypted APFS sparsebundles.

Example shell equivalent:

```bash
hdiutil create \
  -type SPARSEBUNDLE \
  -size 50g \
  -fs APFS \
  -volname AgentKeychain-RegularBrowser \
  -encryption AES-256 \
  ".agent-keychain/volumes/RegularBrowser.sparsebundle"
```

The helper should supply the generated password safely. Avoid putting passwords in process arguments or logs.

## Mounting a Volume

Mount by:

1. Reading the volume password from the selected keychain storage backend.
2. Piping the password to `hdiutil attach -stdinpass`.
3. Passing the expected mountpoint explicitly to `hdiutil attach`.
4. Verifying that the expected image is mounted at the expected mountpoint.

Shell equivalent:

```bash
printf '%s\0' "$PASSWORD" | hdiutil attach "$IMAGE" -stdinpass -mountpoint "$MOUNTPOINT"
```

Do not pass the password as a command-line argument.

Mountpoint rules:

* Create the parent mountpoint path only when needed.
* Resolve and validate the configured mountpoint before use.
* Reject symlink mountpoints.
* After attach, verify the mounted image/device identity, not only that the path exists.
* If the expected mountpoint already exists but is not the configured image mounted by `agent-keychain`, fail closed.

## Locking a Volume

“Locking” means detaching the APFS image:

```bash
hdiutil detach /Volumes/AgentKeychain-qlty-local-agents-RegularBrowser
```

Expose this as `volume lock`.

## v0.1 Volume Lifetime

v0.1 should not implement a background daemon, watcher, or true idle detector.

Simple v0.1 behavior:

* `volume unlock` mounts the volume and exits. The volume stays mounted until `volume lock` or another supervised command detaches it.
* `browser open --detach-on-exit` and `run --detach-on-exit` may detach volumes they mounted after their supervised process exits.
* `agent-keychain` should use one lock file per volume, such as `.agent-keychain/locks/<volume-name>.lock`, to prevent two managed commands from supervising the same volume at the same time.
* If another managed command already holds the volume lock, fail with a clear error instead of trying to share ownership.
* Before detaching, check whether the mountpoint is busy.
* If the mountpoint is busy, do not forcibly detach. Log `volume lock skipped because busy` and leave the volume mounted.
* True idle timeout based on browser or filesystem activity is deferred until a daemon/watch mode exists.

The role field `defaultIdleTimeoutSeconds` may remain in config for forward compatibility, but v0.1 must not present it as an enforced control unless daemon/watch mode is implemented.

# Browser Requirements

## Chrome Launch

Chrome must be launched with a dedicated user data directory inside the mounted encrypted volume.

Example:

```bash
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
  --user-data-dir=/Volumes/AgentKeychain-qlty-local-agents-RegularBrowser/ChromeProfiles/GitHub \
  --no-first-run
```

Do not use `open` for the managed browser command if the process needs to wait for Chrome to exit. Launch the Chrome binary directly using `Process`.

## Browser Profiles

Rules:

* Browser profiles belong to roles.
* Browser profiles must live inside a role-owned encrypted volume.
* A browser profile from one role cannot be opened under another role.
* The human’s normal Chrome profile must never be used.
* The CLI should create profile directories as needed.

Example:

```bash
agent-keychain browser create GitHub \
  --role regular \
  --volume RegularBrowser \
  --reason "Create GitHub browser profile for regular agent work"
```

```bash
agent-keychain browser open GitHub --role regular
```

Privileged example:

```bash
agent-keychain browser open Mercury \
  --role finance \
  --reason "Review payment status for approved contractor invoices"
```

## Privileged Browser Defaults

Roles with higher privileges should be configured with stricter behavior directly in the role profile:

* require reason
* require Touch ID
* detach on exit
* verbose audit logs
* deny environment-variable injection by default

# Role and Policy Requirements

## Role Definition

Each role should support:

```yaml
name: string
description: string
require_reason: boolean
require_touch_id: boolean
default_idle_timeout_seconds: integer
allow_env_injection: boolean
require_dual_approval: boolean
audit_level: normal | verbose
```

There is no separate risk-level abstraction. The policy data lives directly on the role.

## Example Roles

`agent-keychain init` creates a project with no roles. Users define the roles their project needs. Documentation and examples may use roles like these:

```yaml
regular:
  description: "Day-to-day low-risk agent work"
  require_reason: false
  require_touch_id: true
  default_idle_timeout_seconds: 900
  allow_env_injection: true
  require_dual_approval: false
  audit_level: normal

workspace-admin:
  description: "Identity and workspace administration"
  require_reason: true
  require_touch_id: true
  default_idle_timeout_seconds: 300
  allow_env_injection: false
  require_dual_approval: false
  audit_level: verbose

finance:
  description: "Money movement and financial administration"
  require_reason: true
  require_touch_id: true
  default_idle_timeout_seconds: 180
  allow_env_injection: false
  require_dual_approval: false
  audit_level: verbose
```

Dual approval is part of the role model for future use, but v0.1 must not enable it in examples. Full dual-approval enforcement is deferred.

## Enforcement Rules

The CLI must enforce:

1. A secret can only be used by a command running under its owning role.
2. A volume can only be unlocked by a command running under its owning role.
3. A browser profile can only be opened by a command running under its owning role.
4. Roles can require a reason.
5. Roles can disallow raw environment-variable injection.
6. Low-privilege roles must not be able to access privileged resources accidentally.
7. Every privileged operation must create an audit event.
8. Project resources cannot be read from outside the active project unless an explicit project path is provided.

## Policy Mutation Requirements

Policy mutations are privileged operations. This includes:

* `role create`
* `role update`
* `role delete`
* `secret set` when adding or changing secret metadata
* `secret delete`
* `volume create`
* `volume delete`
* `browser create`
* `browser delete`
* config writes that change role policy
* config writes that change resource ownership
* config writes that change keychain service names, volume paths, mountpoints, or browser profile mappings
* config writes that change audit policy or raw-secret exposure policy

Every policy mutation must:

1. Require Touch ID / user-presence.
2. Require a human-readable reason.
3. Validate the resulting config before writing it.
4. Write an audit event with old and new config hashes.
5. Use an atomic config write.
6. Update the config integrity file only after the config write and audit write succeed.

Manual edits to `.agent-keychain/config.json` are not trusted in v0.1. Before any sensitive operation, `agent-keychain` should hash the canonical config and compare it to `.agent-keychain/config.integrity.json`. If the hash does not match, fail closed for secret reads, volume unlocks, browser opens, `run`, and further policy mutation commands until the user explicitly re-trusts the config with Touch ID and a reason.

Atomic config write requirements:

* Hold a project config lock while writing.
* Write canonical JSON to a temporary file in `.agent-keychain/`.
* `fsync` the temporary file.
* Rename the temporary file over `config.json`.
* `fsync` the containing directory.
* Never leave a partially written config as the active config.

Tamper detection is local and best-effort. It is meant to catch accidental or out-of-band config edits before privileged operations, not to withstand malware with the user's filesystem access.

Example rejection:

```bash
agent-keychain run \
  --role regular \
  --secret MERCURY_API_KEY=mercury-api-key \
  -- agent-command
```

Expected error:

```text
Refusing to use secret mercury-api-key from role finance in role regular.
Re-run with --role finance --reason "..."
```

Example rejection:

```bash
agent-keychain browser open Mercury --role workspace-admin
```

Expected error:

```text
Browser profile Mercury belongs to role finance, not workspace-admin.
```

# CLI Specification

## Project Commands

```bash
agent-keychain init [--project-name NAME]
agent-keychain status
agent-keychain config path
agent-keychain config trust-current --reason TEXT
```

Behavior:

* `init` creates the project-local `.agent-keychain` directory, lock directory, config file, config integrity file, audit log, and selected keychain storage backend.
* `status` shows the active project, roles, volumes, mounted state, keychain state, and keychain storage mode.
* `config path` prints the active project config path.
* `config trust-current` re-baselines config integrity after a deliberate manual edit; it requires Touch ID, a reason, and an audit event.

Project discovery:

* Start at the current working directory.
* Walk upward until `.agent-keychain/config.json` is found.
* Allow explicit override with `--project PATH`.

## Audit Log Limitations

The project-local audit log is an accountability aid, not a tamper-proof security log. A process running as the user can delete or rewrite `.agent-keychain/audit.jsonl` and `.agent-keychain/config.integrity.json`.

v0.1 should make local tampering visible where practical by recording config hashes, using hash-chained audit entries, and failing closed for privileged roles when expected integrity state is missing or inconsistent. Strong audit guarantees require a future append-only external sink or brokered service outside the writable project directory.

## Login Keychain Store

Responsibilities:

* in physical-keychain mode, store and retrieve the generated project keychain password
* in fallback mode, store and retrieve project-scoped fallback secret and volume-password items
* require Touch ID / user-presence to retrieve sensitive items
* use `ThisDeviceOnly`

Pseudocode for storing the project keychain password:

```swift
func storeProjectKeychainPassword(service: String, password: String) throws {
    var error: Unmanaged<CFError>?

    guard let access = SecAccessControlCreateWithFlags(
        nil,
        kSecAttrAccessibleWhenUnlockedThisDeviceOnly,
        [.userPresence],
        &error
    ) else {
        throw KeychainError.accessControlCreationFailed
    }

    let query: [String: Any] = [
        kSecClass as String: kSecClassGenericPassword,
        kSecAttrService as String: service,
        kSecAttrAccount as String: "default",
        kSecValueData as String: Data(password.utf8),
        kSecAttrAccessControl as String: access
    ]

    SecItemDelete(query as CFDictionary)

    let status = SecItemAdd(query as CFDictionary, nil)
    guard status == errSecSuccess else {
        throw KeychainError.status(status)
    }
}
```

Pseudocode for reading the project keychain password:

```swift
func readProjectKeychainPassword(service: String, prompt: String) throws -> String {
    let context = LAContext()
    context.localizedReason = prompt

    let query: [String: Any] = [
        kSecClass as String: kSecClassGenericPassword,
        kSecAttrService as String: service,
        kSecAttrAccount as String: "default",
        kSecReturnData as String: true,
        kSecUseAuthenticationContext as String: context,
        kSecUseOperationPrompt as String: prompt
    ]

    var result: CFTypeRef?
    let status = SecItemCopyMatching(query as CFDictionary, &result)

    guard status == errSecSuccess else {
        throw KeychainError.status(status)
    }

    guard let data = result as? Data,
          let value = String(data: data, encoding: .utf8) else {
        throw KeychainError.invalidData
    }

    return value
}
```

## Project Keychain Store

Responsibilities:

* in physical-keychain mode, create the project keychain
* in physical-keychain mode, unlock the project keychain using generated password
* in physical-keychain mode, lock the project keychain
* in physical-keychain mode, store generic password items in project keychain
* in physical-keychain mode, read generic password items from project keychain
* in physical-keychain mode, delete generic password items from project keychain

Implementation options:

* Use Security framework APIs where possible.
* Use `/usr/bin/security` as a fallback if direct APIs are awkward for non-login keychains.

* Never print the keychain password.

## Disk Image Store

Responsibilities:

* create encrypted APFS sparsebundle
* mount sparsebundle
* detach sparsebundle
* check mount status
* check whether mountpoint is busy
* acquire and release per-volume supervision lock files for managed sessions

Create command should:

1. Generate random sparsebundle password.
2. Access the selected keychain storage backend via Touch ID / user-presence.
3. Store sparsebundle password in the selected keychain storage backend.
4. Create image using `hdiutil create`.
5. Use APFS and AES-256 encryption.
6. Store metadata in config.

Mount command should:

1. Validate role access.
2. Access the selected keychain storage backend via Touch ID / user-presence.
3. Read volume password from the selected keychain storage backend.
4. Mount via `hdiutil attach -stdinpass`.
5. Pass the configured mountpoint explicitly.
6. Verify the configured image is mounted at the configured mountpoint.

Pseudocode for attach:

```swift
func attach(imagePath: String, mountpoint: String, password: String) throws {
    let process = Process()
    process.executableURL = URL(fileURLWithPath: "/usr/bin/hdiutil")
    process.arguments = [
        "attach",
        imagePath,
        "-stdinpass",
        "-mountpoint",
        mountpoint
    ]

    let input = Pipe()
    process.standardInput = input

    try process.run()

    input.fileHandleForWriting.write(Data((password + "\0").utf8))
    input.fileHandleForWriting.closeFile()

    process.waitUntilExit()

    guard process.terminationStatus == 0 else {
        throw DiskImageError.attachFailed(process.terminationStatus)
    }

    try verifyMountedImage(imagePath: imagePath, mountpoint: mountpoint)
}
```

Detach command:

```swift
func detach(mountpoint: String) throws {
    let process = Process()
    process.executableURL = URL(fileURLWithPath: "/usr/bin/hdiutil")
    process.arguments = ["detach", mountpoint]
    try process.run()
    process.waitUntilExit()

    guard process.terminationStatus == 0 else {
        throw DiskImageError.detachFailed(process.terminationStatus)
    }
}
```

# Security Requirements

## Must-Haves

* Project-scoped config and state.
* Physical project keychain as the primary storage mode, with explicit fallback keychain mode allowed when the compatibility spike or host runtime checks show physical keychains are not viable.
* In physical-keychain mode, project keychain password generated by the tool, not memorized by the user.
* In physical-keychain mode, project keychain password stored in login keychain with Touch ID / `userPresence`.
* In fallback mode, project secrets and volume passwords stored directly in the login keychain under project-scoped service names with per-item `userPresence`.
* No secret values in config.
* No disk-image passwords in config.
* No passwords or tokens in process arguments.
* No logging of secret values.
* Role ownership enforced before any secret read or volume unlock.
* Browser profiles stored only inside encrypted role-owned volumes.
* Never use the human’s normal Chrome profile.
* Audit events for all sensitive operations.
* Roles can require a reason.
* Roles can deny raw environment-variable injection.
* Privileged browser and run workflows should default to `--detach-on-exit` until true idle timeout enforcement exists.
* In physical-keychain mode, project keychain locked on command exit unless explicitly kept open for a managed run.

## Non-Goals for v0.1

* Cross-platform support.
* Remote secrets management.
* Full tool gateway / MCP gateway.
* Perfect protection against malware running as the user.
* Preventing access while the APFS volume is mounted.
* True idle timeout enforcement without a foreground supervisor.
* Full browser action audit.
* OAuth broker implementation.
* Dual approval implementation.
* Global user-level credential store.

## Known Limitations

* Once a volume is mounted, its contents are plaintext to processes running as the user.
* Once a secret is injected into a child process environment, that process can read it.
* Keychain and Touch ID protect access to stored secrets, not subsequent use of those secrets.
* Chrome and macOS may store some browser-related authentication material outside the managed sparsebundle.
* Chrome profiles can be corrupted if the volume is detached while Chrome is active.
* Browser sessions still carry whatever authority the SaaS account has. Use low-privilege app accounts.

# Future Direction

The long-term direction is to reduce raw credential exposure further.

Instead of:

```bash
agent-keychain run --secret GOOGLE_ADMIN_TOKEN=google-workspace-admin -- agent-command
```

prefer future brokered commands:

```bash
agent-keychain tool google-workspace.add-user \
  --role workspace-admin \
  --email alice@example.com \
  --reason "Onboarding ticket IT-1234"
```

And for finance:

```bash
agent-keychain tool finance.send-payment \
  --role finance \
  --recipient "Acme Security LLC" \
  --amount 4200 \
  --currency USD \
  --reason "Approved invoice INV-1042"
```

In that model, the agent requests a specific action, `agent-keychain` enforces policy, retrieves the credential internally, performs the API call, and logs the structured action. The raw credential is never exposed to the agent.

v0.1 should not implement this, but the project/role/resource/audit model should be designed so brokered tools can be added later.
